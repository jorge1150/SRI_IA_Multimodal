# PlannerAgent — decisión agéntica de estrategia de retrieval vía tool-calling

El sistema multiagente original (`CoordinatorAgent` orquestando `RAGAgent`/`VisionAgent`/`VoiceAgent`/`ResponseAgent`/`TTSAgent`) es una arquitectura multiagente clásica: el coordinador llama a cada agente en una secuencia fija de `if/else`, sin autonomía real — no califica como "software agéntico" en el sentido moderno (LLM decidiendo dinámicamente, no una regla programada). `HybridRetriever.retrieve()` con su modo `"auto"` tampoco decide: cae a `vector_only` por disponibilidad de datos, no por juicio del LLM.

Se agrega `PlannerAgent`: antes del retrieval, le pregunta a Ollama (vía tool-calling nativo — confirmado soportado por Qwen2.5:3b-instruct-q4_K_M, `capabilities: ['completion', 'tools']`) si la consulta necesita GraphRAG además del RAG vectorial (que siempre corre). La decisión es la presencia o ausencia de un único `tool_call` (`buscar_relaciones_grafo`), no un parámetro booleano dentro de la respuesta — más simple y más confiable de parsear con un modelo de 3B.

**Por qué una decisión binaria y no "elegir entre vector y grafo"**: pruebas reales mostraron que el modelo de 3B tiene sesgo fuerte hacia elegir la herramienta vectorial cuando se le ofrecen dos herramientas para elegir entre sí (incluso con prompts diseñados para forzar el uso del grafo). Reformulada como decisión binaria ("¿esta pregunta *también* necesita grafo?", vector siempre corre), la discriminación mejora notablemente (2/3 aciertos en pruebas puntuales vs. prácticamente 0/2 con el framing de elección exclusiva) — es coherente además con cómo ya funcionaba el modo `"auto"` histórico ("vector siempre corre, grafo es el complemento opcional").

**Fallback**: ante cualquier falla (Ollama caído, timeout, respuesta sin tool_calls parseable) degrada a `False` (solo RAG vectorial) — mismo criterio de degradación segura que ya usa `_init_graph_retriever` cuando el grafo no está disponible.

**Activación**: `config.USE_AGENTIC_PLANNER` (default `False`). El chat de producción sigue con el modo `"auto"` fijo hasta validar la confiabilidad del planner con datos reales de `scripts/run_benchmark.py` (que agrega un modo `"agentic"` comparable contra `vector_only`/`graph_only`/`hybrid`, con columna `planning_seconds` propia y `planner_graph_usage_rate` como estadística descriptiva — no una métrica de acierto, no hay ground truth de qué debería haber decidido cada pregunta).

**Complemento visual**: un diagrama de flujo animado en la UI (tab "Consulta Tributaria", botón "Ver Flujo de Agentes") muestra en vivo qué agente está trabajando en cada momento, con el nodo del `PlannerAgent` marcado visualmente como el único punto de decisión real — el resto de nodos son pasos fijos de "coordinador llama, agente ejecuta, devuelve dato", no negociación entre agentes. Es un aporte honesto: no se disfraza el pipeline entero como "conversación entre agentes" cuando en realidad solo un tramo decide algo.
